\## 1. 问题陈述与架构现状



\### 1.1 当前系统定位



系统运行于 NoneBot2 框架之上，采用\*\*双脑协作模型 (Dual-Brain)\*\*，具备：

\- 逻辑脑 (Scheduler)：工具调度与任务路由

\- 演员脑 (Actor)：角色人格渲染与自然语言生成

\- 话题路由 (Topic Router)：四级路由（L1 物理强连通 → L1.5 低熵收容所 → L2 语义向量匹配 → L3 新建话题）

\- 话题收割机 (Topic Harvester)：长期记忆沉淀中枢



\### 1.2 经日志暴露的架构级不协调



通过多用户重度角色扮演 (Multi-User Heavy Roleplay) 场景的压力测试，系统暴露出三个\*\*架构性功能缺失\*\*：



| 异常编号 | 异常名称 | 核心表现 | 根因层级 |

|:---:|:---|:---|:---|

| 一 | 并发语境污染 (Topic Crosstalk) | 两组完全不同的角色扮演对话被吸入同一个 `topic\_id`，形成“缝合怪”上下文 | 路由层 |

| 二 | 强制越权打断 (Aggressive Random Hit) | 7% 概率随机触发器在人类间密集互动时强行插话，破坏沉浸感 | 触发控制层 |

| 三 | 沉浸窗口假阳性销毁 (False Positive Session Destruction) | 用户 `@` 其他人（正常社交行为）被误判为“注意力转移”，暴力销毁沉浸会话 | 会话管理层 |



\### 1.3 演进目标



将系统从\*\*“无状态的随机响应者”\*\*升级为\*\*“基于社交上下文感知的静默陪伴者”\*\*。Bot 不再是在群聊中念台词的笨重机械，而是一个带着实体词“认知力场”在人群中行走、在他人对话时知趣闭嘴、在冷场时轻巧破冰的高情商角色。

\## 2. 设计原则



\### 原则一：启发式优先于计算密集型 (Heuristics over Heavy Compute)



在高频短交互场景下，优先使用确定性规则、滑动窗口统计和轻量级模式匹配，避免引入全局 Embedding 计算带来的 I/O 延迟和算力成本。



\### 原则二：状态机管理生命周期 (FSM over Binary Flags)



将关键状态（会话活跃度、话题存活度）从布尔值升级为有限状态机，支持状态的灵活流转和异步降级，增强系统对群聊高频噪声的鲁棒性。所有状态都必须精确到 `topic\_id` 粒度，严禁跨话题误伤。



\### 原则三：将复杂解耦交给 LLM 原生能力 (Delegate to Attention)



在涉及长文本语义聚类的场景中，不过早引入“多模态会话解缠”等学术界仍在攻坚的复杂算法，而是信任现代 Transformer 架构的自注意力机制，通过强 Prompt 约束让 LLM 进行天然的语义提纯。但在 Phase 1 \& 2 阶段，\*\*绝对禁止\*\*引入任何旁路检测代码，禁止在 Harvester 之外添加 Shadow Context 或离线自愈补丁。



\### 原则四：防御性编程与优雅降级 (Defensive \& Graceful Degradation)



所有外部依赖（Embedding API、数据查询）均设置超时和降级路径。系统在任何异常场景下都能保持基本可用性，静默失败优于崩溃。所有随机触发必须经过硬性逻辑闸门，不可直接执行随机数采样。

\## 3. 核心策略一：Hybrid Routing（结构化锚点 + 实体词软隔离）



\### 3.1 问题针对性



解决\*\*并发语境污染\*\*：多组角色扮演对话在路由层被强行合流。



\### 3.2 设计思路



抛弃原有的单一时间窗口匹配策略，升级为\*\*两层递进式路由决策\*\*：



```

新消息进入

&#x20;   │

&#x20;   ├─ 层级 1：结构化物理逻辑门 (O(1))

&#x20;   │

&#x20;   │   · 若消息存在 Reply 链 → 沿链追溯 topic\_id（现有 L1 逻辑保留）

&#x20;   │   · 若消息包含 @Bot 或唤醒词 → 标记为“主线延续”

&#x20;   │   · 若消息 @ 其他人或无显式关联 → 进入层级 2

&#x20;   │

&#x20;   └─ 层级 2：实体词力场检测 (O(N), 毫秒级)

&#x20;           │

&#x20;           │  为每个活跃 Topic 维护一个极度轻量的“实体词池”

&#x20;           │  (Entity Pool)，通过滑动窗口词频统计维护

&#x20;           │

&#x20;           ├─ 新消息中的【强特征实体】∩ 某 Topic 实体词池 ≠ ∅ → 吸附（软性合流）

&#x20;           └─ 所有 Topic 实体词池均无交集 → 新开支线

```



> \*\*核心约束：\*\* 实体词池仅允许加入【强特征实体】（专有名词、地名、技能名、独特道具等）。所有\*\*通用扮演称谓\*\*（如“姐姐”、“妈妈”、“指挥官”、“主人”）\*\*严禁\*\*加入实体词池，只能用于用户级的 Soft Suspend 冷却重置。



\### 3.3 工程落地细节



\#### 3.3.1 实体词池 (Entity Pool) 的数据结构



```python

@dataclass

class EntityPool:

&#x20;   """每个 Topic 维护一个实体词池，仅保留强特征实体"""

&#x20;   entities: dict\[str, float]  # 强特征实体词 -> 权重（基于词频 + 激进衰减）

&#x20;   last\_updated: float



&#x20;   # 以下词语禁止被加入实体词池

&#x20;   PROHIBITED\_GENERIC\_TERMS: ClassVar\[set\[str]] = {

&#x20;       "姐姐", "妹妹", "哥哥", "弟弟", "妈妈", "爸爸", "主人", "指挥官",

&#x20;       "大人", "前辈", "老师", "长官", "兄弟", "朋友",

&#x20;   }



&#x20;   def add(self, word: str) -> None:

&#x20;       """仅当单词不在禁止列表且长度≥2，且非纯数字时才加入"""

&#x20;       if word not in self.PROHIBITED\_GENERIC\_TERMS and len(word) >= 2:

&#x20;           self.entities\[word] = self.entities.get(word, 0.0) + 1.0



&#x20;   def decay(self, half\_life: float = 120.0) -> None:

&#x20;       """激进衰减：半衰期设为 2 分钟，保证只在紧凑的连贯对话中生效"""

&#x20;       ...



&#x20;   def score\_intersection(self, candidate\_entities: set) -> float:

&#x20;       """计算候选实体词集合与本池的交集加权分"""

&#x20;       ...

```



\#### 3.3.2 实体词提取策略与分级



必须在本地完成，\*\*禁止调用 External API\*\*。提取后按\*\*实体分级\*\*即时分流：



| 类型 | 级别 | 匹配模式 | 用途 | 示例 |

|:---|:---|:---|:---|:---|

| 强特征实体 | \*\*Topic 路由\*\* | 专有名词字典 + 高频首字母大写词 | 加实体词池 | “帕弥什”、“升格网络”、“0号机”、“罗兰” |

| 动作标记 | \*\*Topic 路由\*\* | 正则 `（(.{2,20})）` | 加实体词池 | “（握紧刀柄）”、“（进入0号机）” |

| 数字/ID 标记 | \*\*Topic 路由\*\* | 正则 `\\d{5,}` 等 | 加实体词池 | “JM 下载 350234” |

| 通用扮演称谓 | \*\*仅用户状态\*\* | 特定词典 | 仅用于重置 SOFT\_SUSPEND 冷却 | “姐姐”、“妈妈”、“指挥官”、“主人” |

| 停用词 | \*\*丢弃\*\* | 单字、纯标点、极高频率虚词 | 直接丢弃 | “嗯”、“了”、“的”、“喵” |



> \*\*重要约束：\*\* 实体词池的生命周期必须极短（激进衰减），且只作用于当前连贯对话。一旦对话中断超过衰减半衰期，该池的影响力应迅速归零，防止“过度粘性”导致后续无关对话被误吸附。



\#### 3.3.3 性能约束



\- 实体词提取：纯字符串操作，必须单次 < 1ms

\- 实体词池匹配：遍历当前 `session\_id` 下的所有活跃 Topic 池（上限 10 个），必须单次 < 5ms

\- \*\*禁止外部 I/O\*\*：此层级严禁调用数据库或 API



\### 3.4 与现有 Topic Harvester 的集成



Hybrid Routing 确保 `topic\_id` 在入库阶段就是\*\*语义纯净\*\*的，从而直接提升 Topic Harvester 的提炼质量。

\## 4. 核心策略二：社交熵抑制系统（交互意愿预测）



\### 4.1 问题针对性



解决\*\*强制越权打断\*\*：7% 概率随机触发器在人类间密集互动时强行插话。



\### 4.2 设计思路



本策略实现一个\*\*硬闸逻辑（Hard Gate）\*\*：在执行任何随机数采样之前，必须先通过社交密度检测。只有通过硬闸的消息，才有资格进入概率判定。



```

消息进入（在 random.random() < prob 之前）

&#x20;   │

&#x20;   ├─ 统计最近时间窗口 T 内的发言者集合 S

&#x20;   │

&#x20;   ├─ 若 |S| ≥ 2 且交互频次 > 阈值（高频交互对）：

&#x20;   │      硬闸关闭 → 随机插入概率直接压制为 0，连 0.07 的判定机会都不给

&#x20;   │      Bot 进入“静默读写态”：可记录上下文，严禁输出

&#x20;   │      随后的概率采样步骤被完全跳过

&#x20;   │

&#x20;   └─ 若硬闸未触发（社交冷却期）：

&#x20;          允许进入概率采样，同时评估是否触发破冰逻辑

```



\### 4.3 工程落地细节



\#### 4.3.1 时间窗口队列



```python

@dataclass

class SocialWindow:

&#x20;   events: deque  # deque\[(timestamp, sender\_id)]

&#x20;   window\_size: float = 60.0  # 窗口大小（秒）



&#x20;   def add\_event(self, sender\_id: str) -> None:

&#x20;       ...



&#x20;   def unique\_senders(self) -> int:

&#x20;       return len(set(sender for \_, sender in self.events))



&#x20;   def interaction\_density(self) -> float:

&#x20;       return len(self.events) / self.window\_size

```



\#### 4.3.2 硬闸控制流（强制要求）



系统入口（如 `chat\_entry.py`）中的随机判定代码必须遵循如下结构：



```text

if 是群聊 and 非显式触发(非@Bot、非唤醒词):

&#x20;   if social\_hard\_gate\_check(group\_id):  

&#x20;       # 硬闸触发 → 直接跳过

&#x20;       logger.debug("高频交互对压制：跳过随机插嘴")

&#x20;       return  # 或 finish

&#x20;   else:

&#x20;       # 硬闸未触发 → 继续正常概率判定

&#x20;       prob = group\_cfg.random\_reply\_prob

&#x20;       ...

```



该流程\*\*必须在执行 `random.random() < prob` 之前\*\*完成。



\### 4.4 行为模型



| 场景 | 硬闸状态 | Bot 行为 | 后续概率 |

|:---|:---|:---|:---|

| 群友热聊（≥2人高频） | 关闭 | 静默记录上下文，不输出 | 0% |

| 群聊冷却 | 打开 | 允许概率判定，可能破冰 | 按配置 |

\## 5. 核心策略三：Soft Suspend 有限状态机



\### 5.1 问题针对性



解决\*\*沉浸窗口假阳性销毁\*\*：用户 `@` 其他人的正常社交行为被误判为注意力转移。



\### 5.2 设计思路



将 `is\_active\_session` 从布尔值升级为\*\*基于 `topic\_id` 粒度的三态有限状态机\*\*：



```

&#x20;                显式触发

&#x20;   ┌─────────────────────────────────┐

&#x20;   │                                 ▼

┌───────┐  超时无互动/轻微社交噪声  ┌──────────────┐

│ACTIVE │──────────────────────────→│SOFT\_SUSPEND  │

│       │←──────────────────────────│(半沉浸待机)   │

└───────┘   被唤醒(任何互动)         └──────┬───────┘

&#x20;   ▲                                      │

&#x20;   │                             持续超时/长篇转向

&#x20;   │                                      │

&#x20;   └──────────────────────────────────────┘

&#x20;                         INACTIVE

&#x20;                       (彻底销毁)

```



\*\*强制约束：所有状态必须绑定到 `topic\_id` 粒度。\*\*



\### 5.3 状态转移规则



| 当前状态 | 触发事件 | 目标状态 | 说明 |

|:---|:---|:---|:---|

| `ACTIVE` | 超时无互动 | `SOFT\_SUSPEND` | 自然冷却 |

| `ACTIVE` | 用户 @ 他人（含常规定义的“注意力转移”） | `SOFT\_SUSPEND` | 软性退场 |

| `SOFT\_SUSPEND` | 用户在窗口内再次发言（任意内容） | `ACTIVE` | 立即唤醒 |

| `SOFT\_SUSPEND` | 用户发言包含\*\*通用扮演称谓\*\*或长句 | `ACTIVE` | 通用称谓仅在此处发挥作用 |

| `SOFT\_SUSPEND` | 窗口期超时 | `INACTIVE` | 确认退出 |

| `SOFT\_SUSPEND` | 用户与第三方展开长篇转向 | `INACTIVE` | 加速退出 |



\### 5.4 工程落地细节



数据结构必须重构为\*\*绑定 `topic\_id` 的细粒度层级\*\*：



```python

ACTIVE\_SESSIONS: dict\[int, dict\[str, dict\[str, SessionState]]]

\#                              ↑ group\_id

\#                                      ↑ topic\_id

\#                                               ↑ user\_id



@dataclass

class SessionState:

&#x20;   topic\_id: str            # 强制绑定 topic\_id

&#x20;   status: str              # "ACTIVE" | "SOFT\_SUSPEND" | "INACTIVE"

&#x20;   suspend\_timestamp: float # SOFT\_SUSPEND 的起始时间戳

&#x20;   last\_active: float       # 最后一次互动时间

```



> \*\*关键约束：\*\* `\_cleanup\_sessions` 后台任务必须\*\*精确到 `topic\_id` 级别\*\*去销毁过期状态，严禁因用户在 A 话题退出而误杀其在 B 话题的沉浸态。即清理逻辑应遍历 `topic\_id` 维度，而非仅遍历 `user\_id`。

\## 6. 核心策略四：LLM 原生自愈能力（远期压舱石）



> ⚠️ \*\*工程约束（死命令）：\*\* 本节内容仅作为架构储备和远期参考。在 Phase 1 及 Phase 2 的重构实施中，【绝对禁止】引入任何 Shadow Context 或旁路检测代码。当前阶段唯一的长期记忆入口就是 Topic Harvester。禁止在 Harvester 之外添加任何“自愈补丁”。不要搞自作聪明的额外逻辑，做好基础的 Hybrid Routing 拦截和 FSM 状态管理即可。



\### 6.1 问题针对性



\*\*远期保障\*\*：应对 Hybrid Routing 可能出现漏网的“缝合怪”上下文。



\### 6.2 设计思路



不引入复杂的双峰分布检测算法，而是信任现代大模型的能力边界：



\- \*\*信任一：Self-Attention 的隐式聚类\*\* — Transformer 架构天然具备语义解耦能力。

\- \*\*信任二：JSON Mode + 强效 Prompt 的格式约束\*\* — 让 LLM 自动抛弃混乱的对话脉络，仅提取有价值的结构化设定。



\### 6.3 现有防御已就位



`\_extract\_lore\_from\_topic` 当前的 System Prompt 具备极强的噪声过滤能力。



\### 6.4 触发条件（远期观察后）



仅当 Harvester 草稿在长期观察中质量持续低于预期时，才考虑启动轻量级同质化检测，且必须以独立模块形式存在，不能侵入路由或会话管理的主逻辑。

\## 7. 实施路径与分阶段交付



\### Phase 1：紧急止血（预计 2-3 天）



| 优先级 | 模块 | 实施内容 | 依赖 |

|:---:|:---|:---|:---|

| P0 | 策略二：社交熵抑制（硬闸） | 实现硬闸检测 + 概率判定前置拦截 | 无 |

| P1 | 策略三：Soft Suspend FSM（topic\_id 粒度） | 实现基于 topic\_id 的三态状态机 | 无 |



\*\*交付标准：\*\*

\- 人类间密集互动时，Bot 100% 不会随机插话（硬闸）

\- 用户 @ 其他人后，沉浸会话在 5 分钟内不被误销毁

\- 用户在 A 话题退出绝不误杀 B 话题的状态



\### Phase 2：路由重构（预计 5-7 天）



| 优先级 | 模块 | 实施内容 | 依赖 |

|:---:|:---|:---|:---|

| P0 | 策略一：Hybrid Routing | 实体词分级 + 两层路由决策 + 激进衰减 | Phase 1 |

| P1 | Entity Pool 生命周期 | 强特征实体提取、通用称谓拦截、池大小管理 | P0 |



\*\*交付标准：\*\*

\- 多组角色扮演对话不再合并到同一 `topic\_id`

\- 路由决策不依赖外部 API

\- 通用称谓绝不出现在实体词池中



\### Phase 3：远期优化（观察期后）



| 优先级 | 模块 | 实施内容 | 依赖 |

|:---:|:---|:---|:---|

| P2 | 策略四：旁路校验 | 仅当 Harvester 质量不合格时启动 | 观察数据 |

\## 8. 风险评估与缓解策略



| 风险 | 影响 | 概率 | 缓解策略 |

|:---|:---|:---:|:---|

| 实体词提取规则过于死板 | Routing 精度下降 | 中 | 可配置字典，热更新 |

| 硬闸压制过久，Bot 长期沉默 | 用户感知不活跃 | 低 | 压制最大时长限制，超时自动恢复 |

| Soft Suspend 状态膨胀 | 内存泄漏 | 低 | 后台回收 + 每群最大状态数上限 |

| LLM 自愈能力被高估（但远期才启用） | 草稿质量下滑 | 低 | 观察期 + 质量评分 |
